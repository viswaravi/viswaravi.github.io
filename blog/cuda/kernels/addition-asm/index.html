This post continues my CUDA vector addition series and shifts the focus from performance tuning to compiler output. In the previous post, I looked at how streams and memory behavior affect execution time. Here, I want to take a step back and understand what the CUDA compiler actually generates for a few simple kernel variants.

The goal is not to memorize every PTX instruction or SASS opcode. Instead, the goal is to build intuition:

  • how a small change in the kernel source affects the generated PTX,
  • how PTX maps to lower-level SASS instructions,
  • and how to read the emitted code well enough to connect source-level decisions with GPU behavior.

Why PTX and SASS?

CUDA gives us multiple layers of abstraction:

  • C++ source expresses the algorithm.
  • PTX shows an intermediate representation that is still readable and fairly structured.
  • SASS shows the final machine code executed by the GPU.

Looking at all three together helps answer questions such as:

  • What does the compiler keep as-is?
  • What optimizations does it introduce automatically?
  • Which source-level changes actually matter after compilation?

A quick note on reading assembly

When I first started looking at PTX and SASS, the code felt overwhelming. The trick is to ignore most of the noise at first and look for recurring patterns:

  • kernel arguments and register setup,
  • thread indexing math,
  • memory loads and stores,
  • arithmetic instructions,
  • predicates and branches,
  • and the final return path.

Once those patterns are familiar, even a dense listing becomes much easier to navigate.

Refer to this other post on PTX Instruction Set for a quick refresher on the most common instructions and their syntax.

Kernel variants to keep in mind

To keep the discussion focused, I will compare only small changes to the same core computation:

  • plain vector addition,
  • vector addition with explicit bounds checking,
  • different pointer qualifiers or memory access patterns,
  • and any source tweak that changes the emitted assembly in a meaningful way.

What this post is trying to teach

By the end of this post, I want to be able to answer questions like:

  • Where does the thread index calculation appear in PTX and SASS?
  • How does a simple if (idx < length) show up in the compiled output?
  • What parts of the kernel are optimized away?
  • When do two source variants produce the same assembly, and when do they not?

That is the foundation for learning to read PTX and SASS with confidence.
