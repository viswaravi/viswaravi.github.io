Introduction

When people first learn CUDA memory hierarchy, they usually see a pyramid: registers → shared memory / L1 → L2 → HBM/DRAM. That picture is useful, but it hides the real question:

Which memory structures actually decide whether a warp runs smoothly or stalls?

For most kernels, the first answers are:

  1. the SM-local cache path, where associativity and access pattern decide whether loads are cheap or conflict-heavy, and
  2. the register file, where per-thread register usage decides occupancy, spilling, and how well the GPU can hide latency.

This post is the first part of a memory-hierarchy series. The goal here is not to dive into every cache level in isolation yet. Instead, it is to build the big picture first, then explain two ideas that repeatedly show up in performance work:

  • cache associativity, and
  • register pressure.

Once those two concepts click, a lot of profiler output starts making more sense.

From TPC-Centric Tesla to SM-Centric Modern GPUs

Early Tesla-era diagrams often emphasized the Thread Processing Cluster (TPC) as the visible building block.

  • A TPC grouped SMs together.
  • Texture hardware and parts of the cache path were often shown at that level.
  • This made sense for a graphics-first design lineage.

But for CUDA programmers, modern NVIDIA GPUs are much easier to reason about if you treat the SM as the real center of the machine.

Why?

  1. Warps live at the SM. Scheduling, issue, and latency hiding all happen there.
  2. Registers are allocated at the SM. Register pressure is therefore an SM resource problem.
  3. Shared memory is per SM. So are many of the most important occupancy trade-offs.
  4. Modern L1/TEX behavior is tightly coupled to the SM. NVIDIA tuning guides for Volta, Ampere, and Hopper describe a unified shared-memory / L1 / texture subsystem, with runtime carveout choices.

So the programmer’s mental model should be:

A GPU is a large collection of SMs. Each SM is a throughput-oriented execution engine with its own register resources, warp schedulers, and an SM-local memory subsystem.

The GPU Memory Hierarchy at a Glance

Before discussing associativity and registers, anchor the whole hierarchy:

Level Scope Physically Where Why It Exists Performance Story
Registers Per thread, allocated from SM resources On chip Hold thread-private live values Fastest storage, but limited; too many hurts occupancy
Shared Memory Per thread block / per SM On chip Software-managed reuse and communication Very fast when bank-friendly; programmer controlled
L1 / Texture Cache Per SM or closely coupled to SM resources On chip Catch local/global/texture traffic and help coalescing Fast, but small and throughput-oriented
L2 Cache Chip-wide On chip Shared cache between all SMs and DRAM Last on-chip cache before device memory
Local Memory Per thread address space Usually in device memory Spill target and storage for some thread-private objects Despite the name, it is not on-chip like registers
HBM / DRAM Device-wide Off chip Large backing store High bandwidth, high latency

Two important clarifications for beginners:

  • Local memory is not local to the SM. It is thread-private address space, but physically it has global-memory-like behavior.
  • Shared memory and L1 are related but not identical. On recent architectures they share underlying on-chip resources, but one is programmer-managed while the other is hardware-managed.

CPU Baseline: Why GPU Memory Design Looks So Different

CPU cache design is built around a very different objective:

  • keep a few threads moving,
  • minimize latency for each miss,
  • and spend more hardware budget to reduce conflict misses.

GPU design is built around another objective:

  • keep many warps in flight,
  • tolerate misses through thread-level parallelism,
  • and replicate simpler structures across many SMs.

That is why the same phrase — “cache miss” — means something different emotionally on CPU vs GPU.

  • On a CPU, a miss often directly threatens forward progress.
  • On a GPU, a miss is expected, as long as enough other warps are ready to run.

This difference is the backdrop for both associativity and register allocation.

Cache Associativity: The Minimum Theory You Need

Cache associativity answers one question:

If an address maps to one cache set, how many different lines can live in that set at the same time?

Sets, tags, and ways

A cache is divided into sets. Each address maps to exactly one set. Inside that set, the cache may have one or more candidate slots, often called ways.

So:

  • Direct-mapped cache = 1 way per set
  • 4-way set associative cache = 4 lines per set
  • Fully associative cache = any line can go anywhere

The more ways you have, the easier it is to avoid conflicts. But the lookup logic gets more expensive.

Lookup flow

For any cache access, hardware broadly does this:

  1. Use index bits to choose the set.
  2. Compare the incoming tag against the candidate entries in that set.
  3. If one matches, it is a hit.
  4. Otherwise, it is a miss and one line must eventually be replaced.

In shorthand:

address -> set index -> tag compare(s) -> hit or miss

Why associativity costs hardware

For a 1-way cache:

index -> 1 slot -> 1 compare -> done

For a 4-way cache:

index -> 4 slots
      -> 4 tag compares in parallel
      -> match logic
      -> select returned line

Higher associativity usually means:

  • more comparators,
  • more mux logic,
  • more metadata,
  • more power,
  • and sometimes a longer critical path.

That trade-off is acceptable in CPU designs that optimize hard for miss reduction. GPUs make a different compromise.

Why GPU SM Caches Tend to Be Simpler Than CPU Caches

The key intuition is this:

A CPU often spends extra hardware to prevent a miss. A GPU often spends extra warps to survive a miss.

Design Goal CPU Cache GPU SM-Side Cache
Primary target Single-thread latency Throughput across many warps
Typical response to misses Avoid them aggressively Hide them with parallelism
Associativity pressure Higher Usually lower / simpler
Logic budget Expensive is acceptable Must be replicated broadly

For modern NVIDIA GPUs, official public documentation focuses much more on:

  • cache capacity,
  • unification with shared memory,
  • coalescing behavior,
  • and profiler-visible hit/miss behavior,

than on publishing a stable “the L1 is exactly N-way” rule for every architecture.

That is an important nuance.

Exact associativity and replacement behavior are often architecture-dependent and not always fully disclosed in public vendor docs.

So the safe engineering takeaway is not to memorize one universal associativity number. The real takeaway is:

  • GPU SM-side caches are optimized for throughput,
  • are usually simpler than high-end CPU caches,
  • and still depend heavily on access regularity and locality.

Associativity and Performance: Where It Actually Shows Up

Associativity matters when multiple active cache lines compete for the same set.

That creates conflict misses.

If your access pattern keeps mapping hot data to the same few sets, then even a cache with enough total capacity can behave badly.

A simple example

Imagine a warp repeatedly touches addresses that are spaced in a way that aliases to the same set:

__global__ void stride_conflict(float* out, const float* in, int n, int stride) {
    int tid = blockIdx.x * blockDim.x + threadIdx.x;
    float acc = 0.0f;

    for (int i = tid; i < n; i += blockDim.x * gridDim.x) {
        acc += in[i * stride];
    }

    out[tid] = acc;
}

If stride is unfortunate, the kernel can generate:

  • poor coalescing,
  • low spatial locality,
  • repetitive set conflicts,
  • and more pressure on L2 and DRAM.

In a profiler, the symptom is not “the cache is 4-way.” The symptom is usually one of these:

  • lower L1 hit rate,
  • more sectors per request,
  • more traffic to L2,
  • more long-scoreboard stalls,
  • and less issue efficiency.

That is why associativity is best understood as a behavioral concept, not a trivia fact.

What usually helps

When cache conflicts are hurting, the practical fixes are often:

  1. Improve coalescing so a warp touches fewer cache lines.
  2. Tile into shared memory so data is reused explicitly instead of repeatedly refetched.
  3. Change layout or stride to avoid pathological set reuse.
  4. Reduce working-set churn so hot data actually has a chance to stay on chip.

Registers: The Fastest Memory, and the First Occupancy Limit

If cache associativity explains one side of memory behavior, registers explain the other.

For many CUDA kernels, the most important memory question is not:

“How big is L2?”

It is:

“How many registers did the compiler allocate per thread?”

That is because registers are both:

  • the fastest storage the thread has, and
  • a hard capacity limit on how many warps can reside on an SM.

CPU Registers vs GPU Registers

On CPUs, the register story is dynamic. On GPUs, it is mostly fixed at compile time.

CPU baseline

Conceptually, a CPU can rename architectural registers onto a larger pool of physical registers.

ADD R1, R2, R3
SUB R1, R4, R5

Architecturally, both instructions write R1. Internally, an out-of-order CPU can map them to different physical destinations and keep moving.

That flexibility is one of the reasons CPU register usage feels forgiving.

GPU model

CUDA works differently.

  • PTX exposes virtual registers.
  • ptxas maps them to real hardware registers.
  • That final count becomes part of launch resource usage.

If a kernel uses 96 registers per thread, the SM must reserve that budget for every resident thread of the kernel.

There is no CPU-style dynamic rescue path that suddenly finds more hidden registers at runtime.

What the Register File Looks Like at a High Level

On recent NVIDIA architectures such as Volta, Ampere, and Hopper, public tuning guides document:

  • 64K 32-bit registers per SM, and
  • up to 255 registers per thread.

Nsight Compute’s hardware model also describes the SM as being split into SM sub-partitions (SMSPs). Each SMSP contains its own:

  • warp scheduler,
  • register file,
  • and execution pipelines.

Shared across the SM are larger structures like the unified shared-memory / L1 path.

This is the throughput view of the machine: each sub-partition is trying to keep a pool of resident warps fed with operands and ready to issue.

PTX Registers vs SASS Registers

One source of confusion is that PTX register names are not the final hardware story.

PTX view

ld.global.f32  %f1, [%rd1];
ld.global.f32  %f2, [%rd2];
add.f32        %f3, %f1, %f2;
st.global.f32  [%rd3], %f3;

These are virtual registers. PTX is still an intermediate ISA.

SASS view

Representative final machine code will look more like:

LDG.E R2, [R4]
LDG.E R3, [R6]
FADD R2, R2, R3
STG.E [R8], R2

These R registers are hardware registers from the point of view of the issued machine code.

That mapping matters because register count is what occupancy calculations use, not how pretty the PTX looked.

Why Registers Affect Performance So Strongly

Registers affect performance through three linked mechanisms:

  1. occupancy,
  2. spilling, and
  3. scheduler freedom.

1. Register count limits occupancy

If an SM has $65536$ 32-bit registers and a kernel uses $R$ registers per thread, then the register-limited maximum warps per SM is approximately:

$$ ext{max warps from registers} = \left\lfloor \frac{65536}{32 \times R} \right\rfloor $$

For example:

Registers per thread Theoretical warps from register limit
32 64
64 32
96 21
128 16
255 8

Real occupancy can be lower because block size, shared memory, and per-SM block limits also matter. But the trend is the point:

More registers per thread means fewer resident warps.

And fewer resident warps means fewer opportunities to hide memory latency.

2. Too few registers causes spilling

If the compiler cannot keep thread-private values in registers, some values are spilled into local memory.

That is dangerous because local memory:

  • is thread-private in address space,
  • but lives in device memory,
  • and has similar performance characteristics to global memory.

So spilling often creates exactly the thing you were hoping to avoid: more off-chip traffic and more scoreboard waiting.

3. Register pressure changes code generation

Heavy register pressure can also lead to:

  • more move instructions,
  • shorter live ranges forced by the compiler,
  • less aggressive unrolling,
  • and less instruction-level flexibility.

So register pressure is not just a capacity issue. It changes the shape of the generated kernel.

A Small but Important Nuance: Higher Occupancy Is Not the Final Goal

A common beginner mistake is:

“If fewer registers increase occupancy, then fewer registers must always be better.”

That is not true.

If you force register count too low, the compiler may spill, and the kernel can get slower even if occupancy rises.

The real target is:

enough active warps to hide latency without paying too much spill or recomputation cost.

That is why tuning register usage is usually a balancing act, not a single-direction rule.

What About Register Banks and Operand Collectors?

There is another layer below the simple “more registers good / too many registers bad” story.

Inside the SM sub-partition, the register path is not magic. It has bandwidth limits.

At a high level, modern NVIDIA GPUs use:

  • a register file per SMSP,
  • banked internally,
  • plus buffering / collection logic between the register file and the execution units.

This is often discussed under the umbrella of the operand collector in reverse-engineering and microarchitecture papers.

Why it matters:

  • execution units want operands every cycle,
  • the register file has limited per-cycle read bandwidth,
  • and the machine needs buffering and arbitration to keep the pipelines busy.

For most CUDA developers, though, this is a second-order concern.

The first-order concerns are still:

  1. register count per thread,
  2. spills to local memory,
  3. dependency chains,
  4. and enough active warps to cover memory latency.

Unless you are doing very low-level SASS analysis, those dominate much more often than hand-reasoning about register banks.

How Cache Associativity and Registers Meet in Real Kernels

These topics are usually taught separately, but they collide in practice.

Consider a kernel with:

  • poor global-memory locality,
  • many long-latency loads,
  • and high register usage per thread.

What happens?

  1. The SM-side cache is less effective.
  2. More requests fall through to L2 or DRAM.
  3. Each warp waits longer on memory.
  4. But because register pressure reduced occupancy, there are fewer other warps available to run.

That is the nightmare combination:

bad cache behavior + low occupancy

This is why memory optimization on GPUs is rarely about a single structure in isolation. It is about the interaction between:

  • access pattern,
  • cache behavior,
  • register allocation,
  • and scheduler-visible latency hiding.

What to Look for in Profiling

If this article is the conceptual map, the profiler is where you verify the story.

Nsight Compute

Use Nsight Compute for the kernel-level answer.

Focus on these sections first:

  1. Launch Statistics / Occupancy

    • launch__registers_per_thread
    • launch__occupancy_limit_registers
    • theoretical vs achieved occupancy
    • whether registers, shared memory, warps, or blocks are the limiting resource

  2. Scheduler Statistics

    • active warps vs eligible warps vs issued warps
    • many skipped issue slots usually mean poor latency hiding

  3. Warp Stall Reasons / Source Counters

    • high long_scoreboard often points to waiting on local/global/L1TEX data
    • high lg_throttle can indicate heavy local/global memory pressure
    • high short_scoreboard can be shared-memory or MIO related

  4. Memory Workload Analysis

    • L1/TEX hit rate
    • sectors per request
    • traffic to L2 and DRAM
    • whether accesses look coalesced or scattered

  5. Instruction Statistics / SASS correlation

    • count of local loads and stores
    • register-spilling instructions
    • hot instructions causing memory or dependency pressure

Helpful interpretation patterns:

  • Low occupancy + high long-scoreboard often means register pressure is making cache misses more visible.
  • High local-memory traffic often means spill pressure.
  • High sectors/request often means poor coalescing, which amplifies pressure on L1, L2, and DRAM.

Nsight Systems

Use Nsight Systems for the timeline-level answer.

Look for:

  • whether the kernel is actually the application bottleneck,
  • launch gaps between kernels,
  • overlap or lack of overlap between copies and kernels,
  • whether the GPU is underfed by the CPU,
  • and whether memory migrations or API synchronization dominate the real runtime.

If Nsight Compute says a kernel has cache or register problems, Nsight Systems tells you whether that kernel matters enough to justify deep work.

Visual Profiler / nvvp (legacy)

For older environments, Visual Profiler can still help as a coarse first pass.

Use it to check:

  • occupancy trends,
  • memory throughput trends,
  • and broad stall patterns.

Then confirm the diagnosis with Nsight tools when possible, because the newer tools expose much better source correlation and memory details.

Practical Rules of Thumb

If you want only the working intuition from this article, keep these rules:

  1. Think in SMs, not just in “the GPU.”
  2. Registers are the first memory level to care about for occupancy.
  3. Local memory usually means trouble, because it often means spills.
  4. Cache associativity matters when your access pattern creates conflicts, not because you memorized one hardware number.
  5. GPU caches are throughput helpers, not magical latency erasers.
  6. The goal is balanced latency hiding, not maximum occupancy at any cost.

Closing

Part 1 is about building the mental model:

  • the SM is the center of CUDA performance,
  • cache behavior depends on access regularity and conflict pressure,
  • and registers are both your fastest storage and one of your most important limits.

In the next parts, we can go deeper into the rest of the hierarchy — especially the unified L1/shared path, L2 behavior, and how different memory spaces show up in PTX, SASS, and profilers.
